1. Project Concept Note

1.1 Project Title

OmniDraft — AI-Powered Content Creation Workspace

1.2 Problem Statement / Objective

Professionals and creators routinely juggle multiple tools — one for email drafting, another for summarization, a third for creative writing. This context-switching fragments workflow, reduces productivity, and increases cognitive load. Existing AI writing tools are often single-purpose (e.g., chat-only or summary-only), requiring users to leave one environment for another.

OmniDraft solves this by unifying three core content modes — drafting, summarization, and creative writing — into a single, streamlined workspace powered by a large language model. Users can switch modes instantly, access pre-built templates, upload files, stream AI responses in real time, and persist their conversation history — all without leaving the application.

1.3 Target User & Use Case

User PersonaUse Case
Business professionalsDraft polished emails, proposals, and reports
Students & researchersSummarize articles, papers, and lecture notes
Content creators & writersGenerate blog posts, social captions, and stories
Busy professionalsGet TL;DR summaries of long documents

The primary audience is anyone who regularly produces or consumes text content and wants a single AI-powered assistant to draft, summarize, and create — faster and with less tool-switching.

1.4 LLM Model & API

  • Primary Model: NVIDIA Nemotron (z-ai/glm-5.2) via NVIDIA Build API
  • Fallback Model: Meta Llama 3.1 8B Instruct (meta/llama-3.1-8b-instruct)
  • API Type: NVIDIA Cloud API — Server-Sent Events (SSE) streaming for real-time token delivery
  • Integration: httpx async client with 30-second read timeout, retry logic, and mode-specific system prompts

1.6 Expected User Experience & Outcomes

Users land on a polished marketing page with an interactive mode wheel. After signing in with a single email link (no password), they enter a workspace where they select a mode, optionally choose a template, type their prompt, and watch the AI response stream in word by word. Conversations are automatically saved and accessible from a sidebar. The expected outcomes are:

  • Reduced time spent switching between writing tools
  • Faster first drafts via AI-assisted generation
  • Improved comprehension through instant summarization
  • Creative inspiration through structured prompts and templates
  • A seamless, enjoyable user experience with zero configuration

2. Application Overview & Tech Stack

2.1 Application Overview

OmniDraft is a full-stack, single-page web application consisting of a React frontend, a FastAPI backend, and a Supabase-managed PostgreSQL database. The application is containerized with Docker, served via Nginx reverse proxy, and deployed to AWS Elastic Beanstalk. The frontend communicates with the backend over HTTP (to be upgraded to HTTPS), and the backend in turn communicates with NVIDIA's LLM API over SSE streaming for real-time token generation.

3. Prompting Strategy & Frameworks

3.1 Vibe Coding Methodology

OmniDraft was built entirely through AI-assisted development — "Vibe Coding" as coined by Andrej Karpathy. The development process followed a structured prompt engineering methodology that combined several frameworks:

3.1.1 Role-Goal-Context (RGC) Framework

Every prompt began with a clear role definition, a specific goal, and a rich context block. This eliminated ambiguity and produced high-quality first-pass code.

Pattern: "Act as a [Role]. Build [Goal] using [Tech Stack] with [Constraints]. The user is [Context]."

3.1.2 Markdown-Based Specification Sheets

Rather than natural-language descriptions, each major component was specified using a structured Markdown template with sections for Vibe Profile, Product Brief, Tech Rails, and Design Tokens. This gave the AI unambiguous constraints, reducing hallucinated code by an estimated 30%.

3.1.3 Chain-of-Thought (CoT) Prompting

Complex prompts included step-by-step reasoning instructions: "First, analyze the existing codebase. Second, list the files that need changes. Third, write each file." This reduced logic errors and ensured the AI considered edge cases before generating code.

3.1.4 Prompt Chaining

Instead of one monolithic prompt, development was split into 7 sequential phases, each with its own focused prompt (see Section 4). Each prompt consumed the output of the previous phase, enabling controlled, incremental progress without overwhelming the context window.

5.2 Data Flow

  1. User enters prompt in React frontend
  2. Frontend sends POST /api/chat/stream with mode + message
  3. FastAPI middleware verifies Supabase JWT, applies rate limiting
  4. Router creates/updates conversation in Supabase PostgreSQL, saves user message
  5. LLM Service calls NVIDIA API with mode-specific system prompt + user message
  6. NVIDIA streams tokens back via SSE
  7. FastAPI proxies each token chunk to the frontend via yield
  8. React renders tokens word-by-word using StreamingText with blur animation
  9. After completion, full assistant message is saved to PostgreSQL
  10. User can later view, export, or delete conversations from History

6. Challenges Encountered & Resolutions

6.1 AWS App Runner Subscription Restriction

Challenge: The preferred deployment service (AWS App Runner) was unavailable on the student AWS account, returning a SubscriptionRequiredException error. App Runner requires a support subscription beyond the basic free tier.

Resolution: Migrated to AWS Elastic Beanstalk (Docker AL2023 platform, single-instance t3.micro) as the deployment target. Elastic Beanstalk offered equivalent functionality (container-based deployment, health monitoring, auto-scaling) without the subscription requirement. The same Docker image worked without modification.

6.2 Environment Variable Exposure in Git History

Challenge: During early development, .env files containing Supabase service keys and NVIDIA API keys were accidentally committed to the Git repository and pushed to GitHub. This created a security vulnerability with exposed credentials in version history.

Resolution: Ran git filter-branch to scrub all sensitive files from the entire Git history, removing backend/.env and backend/init_db.py (which contained inline keys). A .gitignore was created to prevent future leaks. The exposed keys were noted for rotation.

6.3 Supabase Auth Redirect Configuration

Challenge: After deployment to Elastic Beanstalk, the Supabase magic link auth flow redirected users to localhost:5173 instead of the live URL, because the Supabase project settings had site_url and uri_allow_list configured only for local development.

Resolution: Updated the Supabase project's authentication settings via the Management API to include the Elastic Beanstalk URL (http://omnidraft-prod.eba-pepktrup.us-east-1.elasticbeanstalk.com) in both site_url and uri_allow_list. The frontend was also updated to use window.location.origin for the emailRedirectTo parameter, making it environment-agnostic.

6.4 SSE Streaming Proxy Buffering

Challenge: The initial Nginx configuration had proxy buffering enabled for SSE streams. This caused the streaming response to buffer in Nginx before being sent to the frontend, negating the real-time streaming benefit.

Resolution: Configured Nginx to disable proxy buffering specifically for the /api/ location block (proxy_buffering off;, proxy_cache off;, proxy_read_timeout 86400s;), allowing SSE chunks to pass through immediately.

6.5 Stuck EB Environment on Initial Setup

Challenge: An earlier attempt at deployment created a stuck Elastic Beanstalk environment (MyIBMWEBSTIE-OMNIDRAFT) that failed to terminate and blocked creation of a new environment.

Resolution: Forcibly terminated the old environment via the AWS CLI (aws elasticbeanstalk terminate-environment) and cleaned up associated resources (EC2 instance, security groups). Created a fresh environment omnidraft-prod with proper configuration.

6.6 AI Hallucination: Imagined Library

Challenge: An early prompt generated code referencing a non-existent Python library for SSE handling. The AI had hallucinated a plausible-sounding library name from its training data.

Resolution: Added explicit constraints to all prompts: "Use only standard library + httpx for HTTP. Do not use any non-standard libraries not listed in the requirements." The SSE implementation was rewritten using FastAPI's native StreamingResponse with httpx async streaming.

7. Key Learnings & Reflection

7.1 Technical Learnings

  • Prompt quality determines output quality. Investing 10 minutes to write a structured, detailed prompt (with role, goal, context, constraints, and examples) produced dramatically better first-pass code than a vague one-line request. The RGC (Role-Goal-Context) framework was consistently effective.
  • Chain-of-Thought reduces debugging. Asking the AI to "think step by step" and "list the files that need changes before writing code" cut the iteration cycle by roughly half. The AI would surface edge cases and dependency issues before generating code, instead of producing broken code that required multiple fix cycles.
  • Context window management is critical. Prompt chaining — splitting development into 7 sequential phases rather than one monolithic prompt — kept each prompt within the model's effective context window and prevented the AI from "forgetting" earlier instructions.
  • Containerization simplifies deployment. The same Docker image that ran locally deployed to AWS Elastic Beanstalk without modification. The "works on my machine" problem was completely eliminated.
  • Security must be proactive, not reactive. Accidentally committing secrets to Git taught the importance of .gitignore setup from day one. A single .gitignore at project root with .env, *.pem, and *key* patterns would have prevented the entire credential exposure incident.

7.2 Vibe Coding Reflection

Vibe Coding fundamentally changes the developer's role from "syntax writer" to "architect and creative director." The most valuable skills were not remembering framework APIs or language syntax, but:

  • Architectural thinking: Designing the system structure, data flow, and component boundaries before prompting
  • Prompt crafting: Writing precise, contextual, constraint-rich instructions
  • Verification: Reviewing AI-generated code for correctness, security, and performance — the AI is a pattern-matcher, not a truth machine
  • Iteration management: Feeding errors back to the AI as learning signals in the "crash and fix" loop
"You are no longer a syntax writer. You are an Architect, a Systems Thinker, a Creative Director." — IBM SkillsBuild Masterclass 1

The project demonstrated that a single developer with no prior FastAPI or Tailwind experience can ship a production-ready, containerized, cloud-deployed AI application in a matter of sessions — provided they master the art of communicating intent to an AI collaborator.
